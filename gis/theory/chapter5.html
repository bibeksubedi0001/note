Part 1: Databases and Spatial DBMS Basics

Textbook Definitions (Textbook pp.158–170)

  • Database: “A large, computerized collection of structured data.” (Textbook p.158)
  • DBMS: “A software package that allows the user to set up, use and maintain a database.” (Textbook p.158)
  • Data Model: “A language that allows the definition of: the structures to store base data, integrity constraints, and computer programs to manipulate data.” (Textbook p.164)
  • Relation (Table): “A collection of tuples (records) that are similarly shaped.” (Textbook p.166)
  • Attribute: “A named field of a tuple, with which each tuple associates a value.” (Textbook p.166)
  • Key: “One or more attributes whose value uniquely identifies a tuple.” (Textbook p.169)
  • Query: “A computer program that extracts data from the database that meet the conditions indicated.” (Textbook p.161)
  • Declarative Language: The user defines what data to extract, not how. The DBMS determines the most efficient way to retrieve it. SQL is a declarative language. (Textbook p.161)

1. What is a Database and a Spatial DBMS?

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. What is a database? What are the reasons for using a database management system? List out the four major types of database models used in GIS [1+3+2] (2071 Magh, Q3a)
  2. What is a spatial DBMS? Explain different types of DBMS structures [2+3] (2073 Magh, Q5)
  • Database: An organized, structured collection of data stored electronically in a computer system, designed for rapid search, retrieval, and updating.
  • Spatial DBMS: A specialized database system engineered specifically to store, query, and manage geographic geometries (Points, Lines, Polygons) alongside traditional tabular data. Standard databases only understand text and numbers; a Spatial DBMS understands coordinates and spatial relationships (like "inside" or "touches").

Reasons for Using a DBMS (Textbook pp.160–162)

  1. Large data volumes: Efficient storage and manipulation of datasets too large for manual handling.
  2. Data integrity: Integrity constraints prevent incorrect data from entering the database (e.g., age cannot be negative).
  3. Concurrent access: Multiple users can operate on the same database simultaneously with different views, without data conflicts.
  4. Declarative queries: Users specify what they want, not how to retrieve it, making data retrieval accessible to non-programmers.
  5. Data model support: Structured frameworks for organizing data logically.
  6. Backup and recovery: Protection against data loss from hardware failure or user error.
  7. Controlled redundancy: Minimizes data duplication while maintaining necessary copies for performance.

2. Reasons for using a Database Management System (DBMS)

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. What is a database? What are the reasons for using a database management system? List out the four major types of database models used in GIS [1+3+2] (2071 Magh, Q3a)

Instead of keeping a bunch of messy Excel spreadsheets (Flat Files), engineers use a DBMS for several critical reasons:

  1. Reduces Data Redundancy: Data is stored in one central location, preventing multiple conflicting copies of the same file.
  2. Maintains Data Integrity: Enforces strict rules (e.g., you cannot enter text into a column meant for numeric Pipe Diameters).
  3. Data Security: Allows administrators to restrict who can view or edit specific data layers.
  4. Concurrent Access: Multiple engineers can view and edit the database simultaneously without crashing the file.

5 Components of a Database System (Lectures p.22)

  1. Hardware: The computer system on which the GIS/DBMS operates (servers, desktops, networked configurations).
  2. Software: The DBMS software that controls and manages the overall database, handling all user requests for data access.
  3. Data: All items stored in the database, organized in rows and columns within tables. Data can be arranged hierarchically, in navigation format, or relationally.
  4. Procedures: The steps to define needs, create a shortlist, evaluate options, implement software, and maintain the database.
  5. Database Access Language: Languages like SQL that include Data Definition Language (DDL) for creating database structures and Data Manipulation Language (DML) for accessing/modifying data.

Flat Files

No direct past exam questions have been asked from this section, but it is part of the syllabus and may appear in future exams.

A flat file is the simplest form of database — a single table stored as a plain text file (CSV, TXT) or spreadsheet (XLS) where all data is contained in one file with rows and columns. There are no relationships between tables, no indexing, and no query language.

Structure of a Flat File

Each row represents one record and each column represents one field/attribute. For example:

Why GIS Moved Beyond Flat Files: Early GIS systems stored attribute data in flat files (e.g., .dbf files accompanying shapefiles). However, as datasets grew and projects required multi-user access, complex queries, and data integrity, GIS adopted relational database management systems (RDBMS). Flat files are still used for simple data exchange (e.g., CSV export from GPS receivers).

Data Model Details (Lectures pp.22–24)

  • Hierarchical Model: Tree structure where each “parent” record has one-to-many “children” through pointers. Access is limited to the master/parent record. A child can only have one parent. (Cho, 1995)
  • Network Model: Solves the hierarchical model’s limitation by allowing many-to-many relationships. A “set” consists of an owner and a member record type. An owner can be a member in another set.
  • Relational Model: Most GIS use this. Key properties: values are atomic, each row is unique, column values are of the same kind, each column has a unique name, sequence of rows/columns is insignificant. Certain fields are designated as keys for indexing.
  • Object-Oriented Model: Based on real-world entities represented as objects with attributes (characteristics) and methods (behaviour). Classes group similar objects. Inheritance allows deriving new classes from existing ones.

1. Hierarchical Database Model

  • Concept: Organizes data in a top-down, tree-like structure. A "parent" record can have multiple "child" records, but a child can strictly only have one parent (a 1-to-Many relationship).
  • Example: A Province (Parent) contains multiple Districts (Children).
  • Disadvantage: Very rigid. If you want to search across different branches of the tree, it is extremely slow. (Draw a simple organization chart tree structure for the exam).

2. Network Database Model

  • Concept: Similar to the hierarchical model, but it allows a child record to have multiple parent records (a Many-to-Many relationship), forming a complex web or graph.
  • Disadvantage: Highly complex to program and maintain. If one link breaks, the whole web can fail.

3. Relational Database Model (RDBMS)

  • Concept: Data is stored in simple, flat 2D tables consisting of Rows (records) and Columns (attributes). Tables are independent but can be mathematically "joined" together using common ID fields.
  • Advantage: Highly flexible, intuitive, and the undisputed standard for modern GIS.

4. Object-Oriented Database Model

  • Concept: Data is stored as "objects" (similar to object-oriented programming like Python or C++). An object contains both the data (attributes) and the executable code (methods/behaviors) that act upon it.

Relational Model Key Concepts (Textbook pp.164–170)

  • Relation Schema: The definition of a relation’s name, its attributes, and their domains. Schemas are stable (change rarely). Relation instances (actual data rows) change frequently.
  • Attribute Domain: The set of atomic values an attribute can take (integer, real, string, date, etc.). “Atomic” means the smallest meaningful unit.
  • SDI (Spatial Data Infrastructure): “The relevant base collection of technologies, policies and institutional arrangements that facilitate the availability of and access to spatial data.” (Textbook p.146)

1. What is an RDBMS and How is it Used in GIS?

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. Explain Relational database system used in GIS [6] (2072 Ashwin, Q2b)

A Relational Database Management System (RDBMS) organizes spatial and non-spatial data into tables (called relations).

  • How it is used in GIS: The GIS software stores the geographic geometry (the shape of a land parcel) in one table and the descriptive attributes (owner name tax value) in another table. It seamlessly links them together behind the scenes allowing you to click a polygon on the map and instantly see its tabular data.

2. How are Relationships Maintained in an RDBMS? (3 Marks)

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. How relationships maintain in Relational Database Management System? [3] (2071 Bhadra Q4b)
  2. How the relationship is maintained? [3] (2073 Bhadra Q5b)
  3. How relationship maintains in relational database management system? [3] (2075 Magh Q4b)

Relationships between separate tables are established and maintained using mathematical keys strictly avoiding hard-coded links.

  • Primary Key: A column in a table where every single row has a unique non-repeating identifier (e.g. a unique Citizen ID number).
  • Foreign Key: A column in a different table that contains values corresponding to the Primary Key of the first table.
  • The Mechanism: By matching the Foreign Key in Table B to the Primary Key in Table A the database temporarily joins the rows together creating a unified view of the data.

3. Why does GIS use RDBMS instead of other database systems? (3 Marks)

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. Describe about RDBMS? Why is it used by GIS instead of other DBMS? [3] (2076 Bhadra Q5a)
  2. Why GIS uses relational database management system? [3] (2079 Chaitra Q5a)
  1. Flexibility: Unlike the rigid hierarchical model RDBMS allows you to join any table to any other table dynamically as long as they share a common key.
  2. Simplicity: Tables (rows and columns) are extremely intuitive for humans to read and understand.
  3. SQL Support: RDBMS uses Structured Query Language (SQL) standardizing how data is queried and edited across almost all software platforms (ArcGIS QGIS PostGIS).

Data Modeling

No direct past exam questions have been asked from this section, but it is part of the syllabus and may appear in future exams.

Data modeling is the process of defining how real-world geographic features and their relationships will be organized, structured, and stored in a GIS database. It bridges the gap between the real world and the digital database.

Levels of Data Modeling

  1. Conceptual Model: A high-level, non-technical description of what data is needed. Uses Entity-Relationship (ER) diagrams.
    • Entities: The real-world objects to be represented (e.g., Road, Building, River, Parcel)
    • Attributes: Properties of each entity (e.g., Road has name, width, surface type)
    • Relationships: How entities relate to each other (e.g., a Building "is located on" a Parcel, a Road "crosses" a River)
  2. Logical Model: Translates the conceptual model into a database structure — defining tables, columns, data types, primary keys, and foreign keys. This is where normalization rules are applied.
  3. Physical Model: The actual implementation in a specific database system (e.g., PostGIS table creation SQL, ArcGIS geodatabase feature classes). Includes decisions about indexing, storage, and performance optimization.

Example: Simple GIS Data Model for a Municipality

Key Principle: Good data modeling ensures that the database can efficiently answer the spatial questions the project demands. A poorly modeled database leads to data redundancy, slow queries, and analytical errors.

Views of the Database

No direct past exam questions have been asked from this section, but it is part of the syllabus and may appear in future exams.

A database can be viewed at three different levels of abstraction, known as the Three-Schema Architecture (ANSI/SPARC model). This architecture ensures data independence — changes at one level do not affect the others.

The Three Schemas

  1. External Schema (View Level): What individual users or applications see. Different users can have different views of the same data. For example, a drainage engineer sees only pipe networks and catchment boundaries, while a land-use planner sees parcels and zoning layers — both accessing the same underlying database.
  2. Conceptual Schema (Logical Level): The community view of the entire database. It describes what data is stored and the relationships between entities, without specifying how the data is physically stored. This is the level defined by the data model (ER diagrams, table structures).
  3. Internal Schema (Physical Level): How the data is physically stored on disk — file structures, indexing methods, storage allocation, and access paths. This is managed by the DBMS and is invisible to end users.

Data Independence

  • Logical Data Independence: The external schema is insulated from changes in the conceptual schema. If a new column is added to a table, existing user views are not affected.
  • Physical Data Independence: The conceptual schema is insulated from changes in the internal schema. If the database is moved to a faster disk or the indexing method changes, the logical structure remains the same.

Why This Matters in GIS: In a large municipal GIS, hundreds of users (engineers, planners, tax officers) access the same spatial database. The three-schema architecture ensures each user gets a customized view without interfering with others, and the database administrator can optimize physical storage without disrupting any user's workflow.

Database Views in Detail (Lectures p.24)

A database view is a subset of a database based on a query that runs on one or more tables. Views are saved as named queries and can be reused.

Types of Views:

  • Dynamic views: Contain data from one or two tables and automatically include all columns. Updated dynamically when related objects change.
  • Static views: Contain data from several tables with specific columns specified in SELECT and WHERE clauses. Updated manually.

Why Views are Important:

  1. Focus users on relevant data, hiding unnecessary information.
  2. Define frequently used joins/projections/selections so users don’t repeat conditions.
  3. Display different data for different users, even when sharing the same database.

Database Normalization

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. What is data Normalization? Explain about 1NF and 2NF [3] (2076 Bhadra Q5b)
  2. What is data normalization? Explain 1NF and 2NF [4] (2078 Chaitra, Q5b)

1. What is Data Normalization?

Data Normalization is a systematic process of organizing data within a relational database to minimize redundancy (unnecessary duplication of data) and ensure data integrity (accuracy and consistency). It involves breaking down large clunky tables into smaller well-structured tables and defining relationships between them using primary and foreign keys.

By normalizing a database a GIS engineer ensures that updating a property owner's name only requires changing it in one single place rather than updating hundreds of separate parcel records.

2. Explain 1NF and 2NF (with rules)

Normalization is done in stages called "Normal Forms."

  • First Normal Form (1NF):
    • The Rule of Atomicity: To be in 1NF every cell in a table must contain a single indivisible (atomic) value.
    • Explanation: You cannot have a column that holds multiple values. For example if a table has a column called "Phone_Numbers" and a single row contains "9841234567, 9801234567" it violates 1NF.
    • The Fix: You must split this data so that each phone number gets its own distinct row and ensure every column has a unique name.
  • Second Normal Form (2NF):
    • The Rule of Full Dependency: To be in 2NF the table must already be in 1NF and it must have no partial dependencies.
    • Explanation: This applies when a table uses a composite primary key (a primary key made of two or more columns combined). Every non-key attribute must depend on the entire composite key not just part of it.
    • The Fix: If your table has a composite key of Project_ID and Engineer_ID but also has a column for Engineer_Name the name only depends on the Engineer_ID not the Project_ID. This is a partial dependency. You must remove Engineer_Name and place it in a separate "Engineers" table.

2. What is the function of a Primary Key in a database table?

PAST EXAM QUESTIONS ON THIS TOPIC:

  1. What is the function of primary key in a table of a database? [3] (2080 Chaitra, Q9b)

The primary key is the absolute most important column in any GIS attribute table (often automatically generated as FID or OBJECTID). You should list these specific functions for full marks:

  1. Guarantees Uniqueness: It ensures that no two rows in a table are exactly identical. Even if two landowners have the exact same name and address, their unique Primary Key (e.g., Citizen_ID) separates them.
  2. Prevents Null Values: A primary key column strictly cannot be empty (NULL). Every single spatial feature must have an identifier.
  3. Anchors Relationships: It serves as the mathematical anchor point to link to other tables. When another table wants to reference this record, it uses the Primary Key as a "Foreign Key," allowing the RDBMS to instantly join the tables together.
  4. Optimizes Searching: Databases automatically "index" primary keys, making spatial queries and data retrieval extremely fast, even with millions of geographic records.

Querying a Database

1. What is meant by querying a database?

Definition: Querying a database means submitting a specific, logically structured request to the Database Management System (DBMS) to retrieve a subset of data that matches certain defined conditions.

In the context of GIS: A query acts as a strict mathematical filter. Instead of scrolling through an attribute table with 753 local level boundaries looking for specific ones, a query automatically searches the database and isolates only the spatial features (polygons) and tabular records that meet your exact engineering or demographic criteria, completely ignoring the rest. Crucially, a query only views or selects the data; it does not permanently delete or alter the original database.

2. SQL Example: Select Rural Municipalities by Population

The Problem Setup:

  • Data Source: A polygon vector table (let's call the table Municipalities).
  • Available Columns: id, name, type-of-municipality, population.
  • The Goal: Find all rows where the municipality is a Rural Municipality ("gaunpalika") AND the population is between 10,000 and 30,000.
